# Developer Onboarding

This page provides a baseline onboarding path for engineers, technical writers, systems architects, and contributors working in the US-SPURS documentation and systems ecosystem.

## Primary Goals

Contributors should understand:

- what US-SPURS is building
- how documentation is organized
- what quality standard is expected
- how to propose, structure, and submit work
- how technical and mission context should shape implementation

## Working Principles

US-SPURS contributors should build with the following assumptions:

- documentation is operational, not ornamental
- systems should be understandable by future contributors
- technical work should map to mission outcomes
- security and resilience are not optional extras
- clear structure beats clever ambiguity

## Recommended First Steps

1. Read the US-SPURS overview page.
2. Review organizational, mission, technology, and infrastructure pages.
3. Review governance and doctrine expectations.
4. Review contribution standards and engineering standards.
5. Confirm the scope of the task before editing content.

## Repository Areas

### `/spurs/`

Primary enterprise documentation set for agency overview, doctrine, mission domains, systems, and infrastructure.

### Root documentation files

Repository-level files such as README, CONTRIBUTING, and standards documents define how contributors work with the repo.

### `.github/`

GitHub-specific process files such as issue templates, pull request templates, and repository automation support files.

## Expected Contribution Types

Contributors may work on:

- documentation expansion
- system architecture definitions
- integration guidance
- engineering standards
- operational playbooks
- workflow and template improvements
- developer tooling and repo hygiene

## Documentation Standards

Contributors should:

- write with precision
- avoid vague placeholders
- prefer operational clarity over marketing language
- organize content so it scales over time
- preserve naming and structural consistency

## Before Opening a Change

A contributor should be able to answer:

- What problem is this change solving?
- Who will use this content or system?
- Where does this belong in the documentation structure?
- What assumptions need to be made explicit?
- Does this conflict with existing doctrine, architecture, or standards?

## Review Expectations

Reviews should examine:

- accuracy
- structural quality
- maintainability
- alignment with mission and doctrine
- technical clarity
- naming consistency
- future extensibility
